Every code change creates some debt. You take on debt to ship faster. That's not bad. That's pragmatic.

What's bad is ignoring it.

## The Types

**Intentional debt** - "We'll clean this up later." You know you're taking on debt. You're choosing to.

**Accidental debt** - Legacy code from before you joined. Frameworks that aged poorly. Technologies that died.

**Cruel debt** - Code written under pressure by people who didn't know better. The kind that makes senior engineers wince.

## When Debt Is Fine

- Prototyping/MVPs
- Time-to-market matters
- You're actively working on the area
- You have a plan to pay it back

## When Debt Becomes A Problem

- Nothing gets faster despite the "simplifications"
- Every change takes longer
- Bugs cluster in specific areas
- Engineers dread touching certain parts
- Testing becomes impossible

## Managing It

**Track it** - Know what debt exists. Don't let it be a surprise.

**Prioritize it** - Not all debt is equal. Some drags everything down. Some is fine to leave.

**Pay it back** - Schedule time. Don't just keep adding features. Regular maintenance prevents rot.

**Prevent it** - Code reviews, testing, good patterns. Prevention is cheaper than cure.

## The Honest Truth

You'll never be debt-free. Accept it. Manage it. Prioritize it.

Some debt is the price of moving fast. Just make sure you're the one deciding when to take it on.
